STEP 01

Python & The Data Stack

Your toolkit before you touch a single model.

Before anything else: Python is the language of AI. Period. Whether it's a scrappy notebook experiment or a production pipeline serving millions of requests, Python is what the industry runs on. If you come from Java, C#, or even JavaScript, Python's simplicity will feel like a relief — but its ecosystem is where the real power lies.

The goal here isn't to become a Python wizard. It's to get comfortable enough that the language stops being the bottleneck. You want to reach the point where when you see a concept, the code writes itself.

The Libraries You Actually Need
🔢
NumPy
N-dimensional arrays. Everything in AI is math — vectors, matrices, tensors. NumPy makes this fast and readable. Every ML library builds on top of it.
🐼
Pandas
Tabular data manipulation. Loading CSVs, cleaning data, filtering rows — you'll use this for every real-world dataset you touch.
📊
Matplotlib / Seaborn
Visualization. Before you train or evaluate a model, you need to see your data. Visualizing distributions, outliers, and patterns is non-negotiable.
⚙️
Scikit-learn
Classic ML in one library. Regression, classification, clustering, preprocessing, pipelines, evaluation metrics — your bread and butter before deep learning.
🌍 Real World Example

Imagine you're building a product recommendation feature for an e-commerce site. Before an LLM ever gets involved, you use Pandas to clean the transaction history, NumPy to compute similarity scores between products, and Scikit-learn to run a quick collaborative filtering model to validate your data quality. This is the groundwork. GenAI sits on top of all of this.

Why devs underestimate this step

If you're coming from software engineering, you might think this is beneath you. Don't. The difference between an engineer who can build with AI and one who just calls APIs is whether they understand the data flowing through the system. When your RAG pipeline returns garbage results, the debug path almost always leads back here — to data quality, embedding shapes, or tensor operations.

STEP 02

ML, DL, CV & NLP Foundations

You need to understand how machines learn before you can make them smarter.

This is where most career-switchers want to skip ahead. Don't. The engineers who truly thrive in GenAI are the ones who understand what's happening inside the black box. You don't need a PhD — but you do need the intuition.

We break this into three sub-topics: how neural networks learn, Computer Vision (CV), and Natural Language Processing (NLP). LLMs are built on all three of these.

How Artificial Neural Networks (ANNs) Work

Think of a neural network as a chain of functions stacked on top of each other. Each layer takes an input, applies a weighted transformation, and passes it to the next. At the end, the network makes a prediction. If it's wrong, the error flows backwards through the network — this is called backpropagation — and each weight is adjusted slightly to make the next prediction a bit better. Do this millions of times and the network "learns."

🧠 The Intuition

Think of it like learning to catch a ball. The first time you try, your brain has random guesses about how to position your hand. Each failed catch gives you feedback. Slowly, your neurons adjust — the connections that led to catching get stronger, the ones that didn't get weaker. That's exactly what gradient descent does to a neural network's weights. The "error" is the ball you missed; backpropagation is your brain figuring out what adjustment to make.

Key Concepts You Must Know
⚖️
Weights & Biases
The learnable parameters. Training = finding the right values for these through repeated error correction.
⚡
Activation Functions
ReLU, Sigmoid, GELU — these introduce non-linearity so networks can model complex patterns, not just straight lines.
📉
Loss & Gradient Descent
Loss measures how wrong you are. Gradient descent is how you walk downhill toward being less wrong. Adam optimizer does this smarter.
🔁
Epochs & Batches
One epoch = one full pass over your dataset. Batching lets you train on chunks rather than everything at once, for speed.
Computer Vision — Why It Matters for GenAI

CNNs (Convolutional Neural Networks) were the breakthrough for image tasks. They scan images in small patches, detect edges, shapes, and finally objects. Today, Vision-Language Models (VLMs) like GPT-4V, Claude, and Gemini use this same idea — they encode images into the same space as text so a single model can reason about both.

Understanding CNNs means you understand the encoder concept — a module that compresses raw data (pixels, words) into a dense, meaningful representation called an embedding. This concept is the spine of every GenAI system.

🌍 Real World Example

At a logistics company, a CV pipeline scans shipping labels from camera feeds. A CNN-based model detects and crops the label, then a VLM (like Gemini) extracts the structured data — name, address, tracking number. The CNN handles "where is the label?" and the VLM handles "what does it say?" Two models, two jobs, one pipeline. This is production AI in 2025.

NLP Before Transformers

Before attention mechanisms, NLP relied on RNNs and LSTMs — networks that processed text one word at a time, left to right. They had a major flaw: they forgot early context by the time they reached the end of a long sentence. This "long-range dependency" problem is exactly what Transformers solved. Understanding why RNNs failed makes you genuinely appreciate why Transformers are so revolutionary.

Also learn: tokenization (how text gets converted to numbers), word embeddings (Word2Vec, GloVe — words as vectors where "king - man + woman ≈ queen"), and TF-IDF for keyword relevance. These aren't legacy knowledge — RAG systems still use TF-IDF for hybrid search.

STEP 03

Transformers & Attention

The architecture that changed everything. Understand this and you understand GenAI.

In 2017, Google published a paper titled "Attention Is All You Need." It introduced the Transformer architecture and within five years, it had replaced almost every other model architecture for language, vision, audio, and more. GPT-4, Claude, Gemini, Llama — all of them are Transformers.

You don't need to implement one from scratch. But you need to understand what attention does, why it's powerful, and what terms like "context window," "tokens," and "layers" actually mean.

Self-Attention — The Core Idea

When you read the sentence "The bank by the river was muddy" — you know "bank" means a riverbank, not a financial institution, because you attended to "river." Your brain relates each word to every other word to understand context.

Self-attention does exactly this. For each word (token), it computes a score against every other token — how relevant is this word to that word? These scores are used to create a weighted blend of all other tokens. This is called the attention score.

🎯 The Spotlight Analogy

Imagine a stage with 10 actors. When actor 3 speaks a line, a spotlight shines on them — but dim lights also fall on actors 1, 5, and 9 because their previous lines gave context. Self-attention is the system that decides where each spotlight falls, for every actor, simultaneously. This is what makes Transformers so much better than RNNs — everything happens in parallel, not one by one.

The Full Transformer Architecture
Key Terms — Demystified
🔑
Tokens
LLMs don't see words — they see tokens. "unbelievable" might be 3 tokens: "un", "believ", "able". 1 token ≈ 0.75 words. Context windows are measured in tokens.
🪟
Context Window
How much the model can "see" at once. GPT-4: 128K. Claude: 200K. Everything outside this window is invisible to the model. This limitation is why RAG exists.
📐
Embeddings
Dense numerical representations of tokens. Words/sentences are mapped to vectors in high-dimensional space. Similar meanings → nearby vectors. The heart of semantic search.
🌡️
Temperature
Controls randomness. Temperature=0: always picks the most likely next token (deterministic). Temperature=1+: more creative but also more likely to hallucinate.
🎭
BERT vs GPT
BERT = encoder-only (reads both directions, good for classification/search). GPT = decoder-only (generates left to right). Most LLMs are GPT-style decoders.
🔧
Fine-Tuning
Taking a pre-trained model and training it further on your specific data. LoRA and QLoRA make this possible without massive compute. It "specialises" the model.
🌍 Real World Example

When you type a message into ChatGPT, your words are tokenized into ~100–300 tokens. These go through 96 Transformer layers (in GPT-4), each applying self-attention — every token attending to every other token in your message. The final layer produces probabilities over the entire vocabulary (~50k words). The token with the highest probability gets picked. Then the whole thing runs again to pick the next token. This happens 30–80 times per second. That's generation.

STEP 04

LLM APIs & Prompt Engineering

The layer between raw model power and real applications.

Unless you're working at a frontier lab, you won't be training LLMs. You'll be calling them. The major providers — OpenAI, Anthropic, Google, Mistral, Groq — all expose their models through APIs. Understanding how these APIs work is where your engineering skills plug directly into GenAI.

What makes this step important isn't just calling the API — any developer can do that. It's understanding the economics, the architecture, and the craft of prompt engineering that separates a GenAI engineer from someone who just wraps GPT in a function.

How LLM APIs Are Structured
# The anatomy of an LLM API call
import anthropic

client = anthropic.Anthropic()

response = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system="You are a senior data analyst. Be precise and concise.",
    messages=[
        {"role": "user", "content": "Summarise this quarterly report: {report_text}"}
    ]
)

# response.content[0].text → your answer
# response.usage.input_tokens + output_tokens → what you pay for
print(response.content[0].text)

Every major provider follows the OpenAI-compatible API standard: a messages array with system/user/assistant roles, a model string, and optional parameters like temperature and max_tokens. Switching from OpenAI to Anthropic to Groq is often just changing the base URL and API key — the code stays identical. This is the OpenAI compatibility layer pattern.

Token Economics — Think Like an Engineer
💰
Input vs Output Tokens
Input (what you send) costs less than output (what the model generates). Prompt caching can reduce input costs by 90% when reusing the same system prompt repeatedly.
⚡
Latency vs Cost
Haiku/Flash = fast & cheap. Sonnet/GPT-4o = balanced. Opus/o3 = expensive & slow but smartest. Pick the right model for the job, not always the biggest one.
🌊
Streaming
Instead of waiting for the full response, stream tokens as they generate. Dramatically improves perceived UX. Essential for any chat-style application.
🔄
Rate Limits & Retries
APIs have TPM (tokens per minute) and RPM (requests per minute) limits. In production, always implement exponential backoff and retry logic around API calls.
Prompt Engineering — It's Actually Engineering

Prompt engineering has a bad reputation because people think it's just "write better instructions." It's not. It's about controlling model behaviour, managing uncertainty, and extracting reliable, structured outputs. Here are the patterns that actually matter in production:

💭
Chain of Thought (CoT)
Ask the model to reason step-by-step before giving a final answer. "Think through this carefully, then answer." Dramatically improves accuracy on complex tasks.
🎯
Few-Shot Prompting
Give 2–5 input/output examples in the prompt. The model pattern-matches to your examples. Especially powerful for structured output formatting.
📋
Structured Output
Ask for JSON, XML, or a specific format. Use the model's JSON mode or tool-calling to guarantee parseable outputs. Never parse freeform text in production.
🛡️
System Prompt Design
The system prompt is your persona and constraint layer. Define role, tone, what the model should/shouldn't do, and how it should handle edge cases.
🌍 Real World Example

A fintech startup uses Claude to extract structured data from messy bank statement PDFs. The system prompt defines the extraction schema. A few-shot prompt shows 3 examples of raw text → JSON output. Chain-of-thought reasoning is disabled (not needed here — just extraction). The output is forced into JSON mode. This single pipeline replaced a team of 5 offshore data analysts. The whole thing runs on <$0.002 per statement.

Frameworks — Don't Over-Engineer

LangChain and LlamaIndex are popular frameworks that add abstraction layers on top of raw API calls. They're useful for prototyping. In production, many teams eventually rip them out and write direct API calls. Learn the frameworks — but always understand what happens underneath. The abstraction should never be magic to you.

STEP 05

RAG, CAG & Knowledge Systems

How you give LLMs a memory — and make them actually useful in production.

Here's the fundamental problem with LLMs: they were trained on data up to a cutoff date. They know nothing about your company's internal docs, your product specs, your customer data, or events that happened last week. Retrieval-Augmented Generation (RAG) is the architecture that solves this — and it's in 80% of real-world GenAI deployments.

Basic RAG — The Core Loop
📚 The Open-Book Exam Analogy

A closed-book exam = pure LLM. The model answers from what it memorised during training. An open-book exam = RAG. Before answering, you search your notes, find the relevant pages, and let those pages inform your answer. RAG gives the LLM a textbook it can look things up in — on the fly.

The Four Types of RAG
Type How It Works Best For Tradeoff
Basic RAG Chunk docs → Embed → Retrieve top-K → Stuff in prompt MVPs, simple Q&A bots Fragile on complex queries
Advanced RAG Hybrid search + reranking + query rewriting + metadata filtering Production knowledge bases More complex to build
CAG Load entire knowledge base into context window (KV-cache) Fixed, small knowledge bases Expensive, not scalable
Agentic RAG Agent decides when, what, and how much to retrieve dynamically Multi-hop reasoning, research Slower, harder to debug
CAG — Cache Augmented Generation

CAG is the new kid on the block. Instead of retrieving relevant chunks at query time, you preload your entire knowledge base into the model's context window (using KV-cache), then ask questions. The model "sees" everything simultaneously. This sidesteps chunking, embedding, and retrieval entirely — but only works if your knowledge base is small enough to fit (under ~200K tokens) and doesn't change frequently. Think of it as RAG with the retrieval step replaced by "just give it everything."

Advanced RAG — What Real Production Looks Like
🔍
Hybrid Search
Combine semantic (vector) search with keyword (BM25/TF-IDF) search. Semantic catches meaning; keyword catches exact terms. Together they cover almost all query types.
📊
Reranking
After retrieval, a cross-encoder reranker re-scores the top 20 results to pick the best 5. Cohere Rerank, FlashRank — dramatically improves relevance.
✍️
Query Rewriting
Use an LLM to rephrase the user's query before searching. "What does GDPR say about cookies?" → "GDPR article 6 lawful basis cookie consent requirements"
🗄️
Vector Databases
Qdrant, Pinecone, Weaviate, ChromaDB, pgvector. Store embeddings and enable fast approximate nearest-neighbor (ANN) search across millions of vectors.
✂️
Chunking Strategy
How you split documents matters enormously. Fixed-size, sentence-based, semantic, or hierarchical chunking. Wrong chunking = broken retrieval, even with perfect embeddings.
🎯
Evaluation
RAGAS framework measures retrieval quality (context recall, context precision) and generation quality (faithfulness, answer relevance). Never ship RAG without evals.
🌍 Real World Example — Enterprise Support Bot

A SaaS company builds a support bot that can answer questions from 10,000+ pages of documentation. Basic RAG: embed all docs into Qdrant, retrieve top-5 chunks, answer. Works for simple questions. Problem: "What's the difference between Plan A and Plan B features?" requires info from two different doc sections — basic RAG returns chunks about one plan, misses the other. Advanced RAG fix: query rewriting breaks it into 2 sub-queries, hybrid search finds both sections, reranker picks the best 5 combined chunks, model compares them accurately. Deflection rate goes from 34% → 71%.

STEP 06

Agentic & Multi-Agent Systems

The frontier. AI that doesn't just answer — it thinks, acts, and coordinates.

Every previous step was about making LLMs smarter, faster, or better-informed. This step is about making them autonomous. An agent doesn't just respond to a query — it plans a sequence of steps, uses tools to interact with the world, observes results, and continues until the goal is complete. This is where GenAI stops being a fancy chatbot and becomes actual software that does work.

What Is an AI Agent?
🤖 The Intern Analogy

A basic LLM is like a brilliant friend who can answer any question — but only if you bring them exactly the right information. An agent is like a brilliant intern with a computer. You give them a goal: "Research our top 3 competitors and summarise their pricing." They don't wait for you to hand them data. They search the web, open URLs, read pages, compare, and write the summary. Tools are the computer. The LLM is the brain. The agent loop is what keeps them working until the job is done.

The Agent Loop (ReAct Pattern)
User Goal
→
LLM Reasons
→
Pick Tool
→
Execute Tool
→
Observe Result
→
Reason Again
→
... until done

This loop is called ReAct (Reason + Act). The model reasons about what to do next, takes an action (calls a tool), observes the result, reasons again, and repeats. The LLM is in the driver's seat — it decides when it has enough information and when to stop.

Tools — How Agents Interact with the World
🔍
Web Search
Search the internet for current info. Tavily, SerpAPI, Brave Search API. Essential for any agent doing research or needing real-time data.
💻
Code Execution
Run Python in a sandbox, do math, generate charts, process files. The agent writes code; the code runs; results go back to the agent. Powerful and risky — sandbox it.
🗄️
Database / API Calls
Query your SQL DB, call REST APIs, read/write files. This is how agents integrate with existing systems — connecting AI to your actual business data.
📧
Communication Tools
Send emails, Slack messages, create calendar events. When an agent can communicate with humans and systems, it becomes a genuine workflow participant.
MCP — Model Context Protocol

MCP (by Anthropic) is the emerging standard for connecting AI agents to external tools and data sources. Think of it as a universal plugin system — instead of hardcoding each tool integration, you define MCP servers that expose capabilities. The agent discovers and calls them dynamically. It's becoming the USB-C of AI tooling — one standard to connect everything. If you're building agentic systems in 2025, you need to know MCP.

Multi-Agent Systems — When One Agent Isn't Enough

Some tasks are too complex, too long, or too multi-disciplinary for a single agent. Multi-agent systems break work into specialised agents that collaborate. There's typically an orchestrator agent (the manager) that plans the work and delegates to worker/sub-agents (the specialists).

🎯
Orchestrator Agent
The "project manager." It receives the high-level goal, breaks it into tasks, assigns tasks to specialist agents, and synthesises results into a final output.
🔬
Specialist Agents
Each agent is optimised for one job: Research Agent, Writing Agent, Data Analysis Agent, Critic Agent. Smaller scope → better reliability → more predictable outputs.
🔁
Human-in-the-Loop
Pause for human approval at critical decision points. Prevents runaway agents making expensive or irreversible mistakes. A checkpoint, not a weakness.
🌐
LangGraph / Agno
Frameworks for building stateful, multi-step agent graphs. LangGraph gives you fine-grained control over agent flow. Agno (formerly Phidata) is simpler and production-first.
🌍 Real World Example — Financial Analysis System

An Orchestrator agent receives: "Analyse Q3 performance for our top 5 portfolios and flag any risks."

It spins up: a Data Retrieval Agent (pulls numbers from the database), a Market Research Agent (searches for macro events during Q3), a Risk Analysis Agent (runs quantitative risk models via code execution), and a Writer Agent (drafts the final report). The orchestrator synthesises all outputs, pauses for a human to approve the risk flags, then sends the final report to Slack. What would take an analyst 4 hours takes the system 8 minutes. This is a real architecture teams are shipping today.

Guardrails — Don't Skip This

Agents are powerful and that makes them dangerous if unchecked. In production, you always need: input validation (prevent prompt injection), tool sandboxing (code execution in isolated containers), output validation (check model outputs against expected schemas), cost caps (limit tokens/API calls per run), and audit logs (log every tool call for debugging). A runaway agent calling APIs in a loop can burn through budget in minutes.

// LangGraph Implementation
The StateGraph & Parallel Execution

How LangGraph wires all of this together — the actual code pattern.

LangGraph uses a StateGraph — a directed graph where each node is an agent function and edges define the flow. The key power here is parallel fan-out: using Send() primitives, you can dispatch all 4 teams simultaneously and wait for all to complete before proceeding. This cuts execution time from ~20 minutes sequential to ~4 minutes parallel. 
from langgraph.graph import StateGraph, START, END
from langgraph.types import Send
from typing import TypedDict, Annotated
import operator

# ─── Shared State — every agent reads/writes this ───
class StartupAnalysisState(TypedDict):
    startup_idea:    str
    task_graph:      dict                              # orchestrator output
    market_output:   Annotated[list, operator.add]  # parallel list merge
    financial_output:Annotated[list, operator.add]
    strategic_output:Annotated[list, operator.add]
    rag_output:      Annotated[list, operator.add]
    critique:        dict
    scores:          dict
    final_report:    str

# ─── Orchestrator decides which teams to dispatch ───
def orchestrator_node(state: StartupAnalysisState):
    task = parse_startup_idea(state["startup_idea"])
    # Returns Send() objects for parallel dispatch
    return [
        Send("market_team",    {"task": task, "focus": "market"}),
        Send("financial_team", {"task": task, "focus": "financial"}),
        Send("strategic_team", {"task": task, "focus": "strategic"}),
        Send("rag_knowledge",  {"task": task, "focus": "frameworks"}),
    ]

# ─── Build the graph ───
graph = StateGraph(StartupAnalysisState)

graph.add_node("orchestrator",   orchestrator_node)
graph.add_node("market_team",    market_team_node)     # runs 3 agents in sequence
graph.add_node("financial_team", financial_team_node)
graph.add_node("strategic_team", strategic_team_node)
graph.add_node("rag_knowledge",  rag_team_node)
graph.add_node("synthesis_gate", synthesis_gate_node) # aggregates all 4
graph.add_node("devils_advocate",devils_advocate_node)
graph.add_node("india_scorer",   india_scorer_node)
graph.add_node("global_scorer",  global_scorer_node)
graph.add_node("report_architect",report_architect_node)

# ─── Edges ───
graph.add_edge(START, "orchestrator")
graph.add_conditional_edges("orchestrator", orchestrator_node)  # fan-out via Send()

# all teams → synthesis gate (auto-joins when all complete)
for team in ["market_team", "financial_team", "strategic_team", "rag_knowledge"]:
    graph.add_edge(team, "synthesis_gate")

# synthesis → 3 critique agents (another parallel fan-out)
graph.add_conditional_edges("synthesis_gate", lambda _: [
    Send("devils_advocate", {}),
    Send("india_scorer", {}),
    Send("global_scorer", {}),
])

for node in ["devils_advocate", "india_scorer", "global_scorer"]:
    graph.add_edge(node, "report_architect")

graph.add_edge("report_architect", END)

app = graph.compile()
result = await app.ainvoke({"startup_idea": user_input})

Why Annotated[list, operator.add] in State? When multiple teams write to the same state key simultaneously in parallel, LangGraph needs to know how to merge them. Annotated[list, operator.add] tells LangGraph to concatenate lists instead of overwriting. This is the reducer pattern — exactly how Redux reducers work, applied to distributed agent state.

The Knowledge Layer — Where RAG Lives

Agentic RAG Pattern — Multi-Hop Retrieval
class AgenticRAGAgent:
    def __init__(self, qdrant_client, collection: str):
        self.qdrant = qdrant_client
        self.collection = collection
        self.llm = ChatOpenAI(model="gpt-4o")

    async def retrieve(self, task: dict, max_hops: int = 3) -> list[str]:
        context_chunks = []
        query = task["startup_idea"]

        for hop in range(max_hops):
            # Step 1: Agent formulates a targeted retrieval query
            retrieval_query = await self.llm.ainvoke([
                SystemMessage("Generate a precise search query for this knowledge base."),
                HumanMessage(f"Current context: {context_chunks}\nNeed more info about: {query}")
            ])

            # Step 2: Semantic search in Qdrant
            results = self.qdrant.search(
                collection_name=self.collection,
                query_vector=embed(retrieval_query.content),
                query_filter=Filter(must=[FieldCondition(
                    key="domain", match=MatchValue(value=task["domain"])
                )]),
                limit=8,
                with_payload=True
            )
            context_chunks.extend([r.payload["text"] for r in results])

            # Step 3: Agent evaluates if retrieval is sufficient
            sufficiency = await self.llm.ainvoke([
                HumanMessage(f"Context retrieved: {context_chunks}\nIs this enough to answer: {query}? Reply YES or NO")
            ])
            if "YES" in sufficiency.content:
                break   # ← Agentic decision: stop retrieving

        return context_chunks  # pass to analysis agent
// MCP Integration
Model Context Protocol — 3 Servers

MCP is the plugin system that connects agents to external capabilities.

Instead of hardcoding API keys and HTTP calls inside each agent, MCP servers expose standardised tools that any agent can discover and call. When you add a new MCP server, all agents can use its tools immediately — no code changes. This is why MCP is called the "USB-C of AI tooling."
MCP Server 01
Tavily Search MCP
Real-time web search with grounded, summarised results. Used by Market, Financial, and Trend agents. Returns structured JSON with source citations — agents can verify claims.
tavily_search tavily_extract news_search
MCP Server 02
Filesystem MCP
Agents can read/write files — save intermediate outputs to disk, load reference documents, write the final report as a formatted PDF. Prevents re-computation on partial failures.
read_file write_file list_directory create_directory
MCP Server 03
Python Executor MCP
Sandboxed code execution. Unit Economics Agent writes Python to model financial scenarios. Market Sizing Agent runs calculations. Output is captured and fed back into agent state.
execute_python install_package read_output
MCP Server 04 · Optional
Crunchbase MCP
Direct access to Crunchbase's startup database. Investment Scout Agent retrieves funding rounds, investor details, valuation multiples, and comparable exits without scraping.
search_companies get_funding_rounds get_investors
MCP Server 05 · Optional
Qdrant MCP
Exposes Qdrant operations as MCP tools. Agents can query collections, upsert new documents, and update metadata — turning the vector DB into a living knowledge base that improves over time.
qdrant_search qdrant_upsert list_collections
MCP Server 06 · Optional
Slack / Email MCP
Human-in-the-loop checkpoint. When Devil's Advocate flags a critical risk, Report Architect sends a summary to the founder's Slack/email before generating the final report — waiting for approval.
send_message wait_for_reply create_thread

Phased Build Plan — 8 Weeks

Don't build everything at once. Ship value incrementally, learn from each phase.

Phase 1 · Week 1–2
Single-Agent Proof of Concept
Build one agent — the Market Sizing Agent — with Tavily search and structured Pydantic output. Validate the basic RAG loop with one Qdrant collection (india_market_reports). Get a working end-to-end call from startup idea to market analysis JSON.
Phase 2 · Week 3–4
Build All 5 Teams + Sequential Flow
Add all remaining agents. Wire them sequentially (not parallel yet) in LangGraph. Index all 5 Qdrant collections. Validate each team's output quality independently. Add Pydantic schemas for every agent output. Set up LangSmith tracing now — you'll need it.
Phase 3 · Week 5
Parallelise + State Management
Switch sequential team execution to parallel using LangGraph's Send() primitive. Add the Annotated reducer pattern to State. Test with 5 different startup ideas. Measure end-to-end latency before and after parallelisation. Implement rate limit handling with asyncio Semaphore.
Phase 4 · Week 6
AgenticRAG + Multi-Hop Retrieval
Upgrade the RAG team from basic retrieval to Agentic RAG with the multi-hop loop. Implement hybrid search (dense + BM25) in Qdrant. Add a reranker (Cohere or FlashRank). Run RAGAS evals on retrieval quality before and after — document the improvement.
Phase 5 · Week 7
MCP Integration + Human-in-the-Loop
Replace hardcoded API calls with MCP tool calls. Add the Slack MCP for the human-in-the-loop checkpoint at the Devil's Advocate stage. Add the Python Executor MCP for Unit Economics calculations. Test fault tolerance — what happens if one MCP server is down?
Phase 6 · Week 8
FastAPI + Evaluation + GitHub Docs
Wrap the app in FastAPI with a streaming endpoint. Write a comprehensive evaluation pipeline: RAGAS for RAG quality, LLM-as-judge for report coherence, schema validation tests for every agent. Write the README covering architecture decisions, benchmarks, and failure modes. This is your portfolio piece.
